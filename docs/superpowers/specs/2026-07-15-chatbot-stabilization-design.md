# LocalHub 챗봇 안정화 및 통합 검색 설계

## 목표

현재 Vue 3 챗봇 UI와 Netlify Function 구조를 유지하면서 자연어 검색 정확도, 요청 안정성, 후속 질문 처리, 비용 효율을 개선한다. 챗봇 범위는 서울 권역의 제공 JSON 전체와 익명 커뮤니티 게시글로 확장하고, 프로젝트와 관련된 가벼운 일상대화를 허용한다.

## 확정된 과제 제약

- 프론트엔드는 Vue.js 3와 Vite를 사용한다.
- 챗봇의 OpenAI 호출은 Netlify Functions에서 수행한다.
- `OPENAI_API_KEY`는 서버 전용 `.env` 및 Netlify 환경변수로 관리한다.
- API 키에 `VITE_` 접두사를 붙이거나 브라우저 번들에 포함하지 않는다.
- 제공 JSON을 우선 사용하고, 추가 데이터는 출처와 라이선스를 기록한다.
- 선택 기능으로 구현된 Leaflet 지도는 유지한다.
- 새로운 프레임워크나 데이터베이스는 추가하지 않는다.

## 현재 구조

```text
FestivalChatWidget.vue
  -> POST /api/chat
  -> netlify/functions/chat.mts
  -> festival-search.ts
  -> 관련 축제 최대 5건
  -> OpenAI Responses API 또는 검색 기반 fallback
```

이 구조는 API 키를 브라우저에 노출하지 않고, LLM 호출 전에 제공 데이터를 선별한다는 점에서 유지할 가치가 있다.

## 확인된 문제와 원인

### 검색 정확도

`extractTerms()`가 공백 단위 토큰에서 조사와 날짜 표현을 제거하지 않는다. 따라서 다음과 같은 표현이 데이터 필드에 존재하지 않는 검색어로 남는다.

- `9월에 축제`: `9월에`
- `무료인 축제`: `무료인`
- `종로구에서 열리는 축제`: `종로구에서`, `열리는`
- `2026년 9월 축제`: `2026년`

날짜 의도 분석도 `오늘`, `주말`, `이번 달`, 단순 `N월`만 지원하여 `내일`, `이번 주`, `다음 달`, `YYYY년 M월`을 처리하지 못한다.

### 대화 문맥

프론트는 메시지를 화면에 보관하지만 API 요청은 현재 질문 하나만 전송한다. 서버도 이전 결과를 알지 못하므로 `그중 무료인 것만`, `첫 번째 장소는 어디야?` 같은 후속 질문을 독립 질문으로 오해한다.

### 요청 안정성

브라우저 요청에 시간 제한이 없다. 요청이 끝나지 않으면 `isLoading`이 계속 `true`라서 추가 전송이 막힌다. 실패 시 사용자 질문은 대화에 남지만 같은 질문을 다시 보내는 명시적 동작이 없다.

### 비용과 지연

응답 지시는 8문장 이내지만 `max_output_tokens`는 2,000이다. 이 값은 추론 토큰도 포함하므로 임의로 크게 줄이지 않고 대표 질의 평가로 적정값을 정해야 한다. 단순 목록·일정 질의는 이미 검색 기반 답변으로 완성할 수 있으므로 LLM 호출 여부를 의도별로 구분할 여지가 있다.

### 기능 범위

현재 검색 공급자는 축제 201건만 지원한다. 과제의 지역정보 챗봇 목표를 충족하려면 관광지, 문화시설, 레포츠, 숙박, 쇼핑, 여행코스와 익명 커뮤니티 게시글을 단계적으로 연결해야 한다. 제공 폴더에 음식점 JSON이 없으므로 음식점 검색을 완료 기능으로 표시하지 않는다.

## 권장 아키텍처

### 1단계: 축제 챗봇 안정화

HTTP 계약은 작은 범위에서 확장한다.

```ts
interface FestivalChatRequest {
  question: string
  context?: {
    recentMessages: Array<{
      role: 'user' | 'assistant'
      content: string
    }>
    sourceIds: string[]
  }
}
```

- `recentMessages`는 최근 2개 대화 왕복만 전달한다.
- 각 메시지는 서버에서 길이와 개수를 다시 제한한다.
- `sourceIds`는 직전 답변에서 노출된 항목만 포함하고 최대 5개로 제한한다.
- 서버는 후속 표현이 감지된 경우에만 직전 항목을 후보 집합으로 사용한다.
- 일반 질문은 기존 전체 축제 데이터 검색을 사용한다.

검색기는 다음 책임으로 분리한다.

```text
normalizeKoreanQuery -> parseDateIntent -> parseFilters -> extractSearchTerms -> rankResults
```

- 조사 제거는 알려진 의도 토큰과 자치구·날짜 표현에 한정한다.
- 일반 고유명사의 마지막 글자를 무조건 제거하지 않는다.
- 날짜는 서울 시간 기준으로 계산한다.
- 명시적 연도 없이 지난 월을 질문하면 현재 연도를 기본값으로 유지한다. 자동으로 다음 해를 추측하지 않는다.

요청 안정성은 다음과 같이 처리한다.

- 프론트 요청 제한시간: 15초
- 제한시간 초과 시 전용 오류 문구와 `다시 시도` 버튼 표시
- Netlify Function의 OpenAI 요청 제한시간: 12초
- OpenAI SDK 자동 재시도: 최대 1회
- 컴포넌트 해제 시 진행 중 요청 취소
- 성공, 입력 오류, Rate Limit, 시간 초과, 서버 오류를 서로 다른 오류 코드로 표시

### 2단계: 통합 지역정보 검색

공통 검색 결과 계약을 도입한다.

```ts
type LocalHubSourceType =
  | 'festival'
  | 'attraction'
  | 'culture'
  | 'leisure'
  | 'accommodation'
  | 'shopping'
  | 'course'
  | 'community'

interface LocalHubSource {
  id: string
  type: LocalHubSourceType
  title: string
  summary: string | null
  address: string | null
  imageUrl: string | null
  longitude: number | null
  latitude: number | null
  metadata: Record<string, string | number | boolean | null>
}
```

- 데이터 유형별 원본 파서는 독립 모듈로 둔다.
- 모든 파서는 `LocalHubSource`로 변환한 뒤 공통 랭커에 전달한다.
- 질문에서 유형이 명확하면 해당 공급자만 검색한다.
- 유형이 불명확하면 비용이 낮은 키워드 검색으로 후보를 만든 뒤 최대 5건만 LLM에 전달한다.
- 커뮤니티가 `localStorage` 기반이면 브라우저가 관련 게시글의 제목·본문 일부만 요청에 포함한다. 비밀번호는 절대 전송하지 않는다.
- 음식점 데이터가 제공되기 전에는 지원 유형과 안내 문구에서 제외한다.

### 질문 범위 라우팅

질문은 LLM 호출 전에 세 범주로 분류한다.

```text
project-data -> 관련 데이터 공급자 검색 -> LLM 근거 답변
project-chat -> 데이터 검색 없이 LLM 일상대화
out-of-scope -> 지원 범위 안내
```

- `project-data`에는 축제, 관광지, 문화시설, 레포츠, 숙박, 쇼핑, 여행코스, 커뮤니티 질문이 포함된다.
- `project-chat`에는 인사, 감사, 작별, 챗봇 사용법, 서울 여행 취향 대화, 직전 답변에 대한 감상이 포함된다.
- 실시간 날씨, 의료·법률·투자 조언, 프로젝트와 무관한 작업 요청은 `out-of-scope`로 처리한다.
- 범주 판정은 우선 명시적 키워드와 대화 상태를 사용하고, 애매한 질문은 `project-chat`으로 처리하여 자연스러운 응답을 허용한다.

### 응답 속도 최적화 결정

이번 변경에서는 SSE 스트리밍을 도입하지 않는다. 실제 응답 시간을 줄이는 다음 조합을 우선 적용한다.

1. 질문 유형에 맞는 공급자만 검색한다. 전체 6,518건을 매 요청마다 통합 순회하지 않는다.
2. 공급자별 정규화 검색 인덱스를 모듈 초기화 시 한 번 생성하고 warm invocation에서 재사용한다.
3. 명확한 단일 유형 질문은 후보 3건, 복합·모호한 질문은 최대 5건만 LLM에 전달한다.
4. `gpt-5.6-luna`에 `reasoning.effort: 'none'`을 사용한다. 품질 회귀가 확인되면 해당 요청 유형만 `low`로 되돌린다.
5. `max_output_tokens`는 800으로 제한한다.
6. OpenAI 요청 제한시간은 12초, 브라우저 전체 요청 제한시간은 15초로 제한한다.
7. 프로젝트 관련 일상대화는 JSON 검색과 근거 직렬화를 생략하고 바로 LLM에 전달한다.
8. 정상적인 지원 범위 질문은 기본적으로 LLM을 사용하며, fallback은 API 키 없음·시간 초과·OpenAI 오류에만 사용한다.

이 조합은 스트리밍보다 변경 범위가 작고, 검색·추론·출력 세 구간의 실제 처리량을 함께 줄인다. 스트리밍은 위 최적화 이후에도 체감 지연이 큰 경우 별도 단계로 검토한다.

## OpenAI 연동 정책

- `gpt-5.6-luna`와 Responses API 사용은 유지한다.
- 단순 근거 기반 질의와 프로젝트 관련 일상대화는 `reasoning.effort: 'none'`을 기본값으로 사용한다.
- `max_output_tokens`는 800으로 제한한다.
- `store: false`를 유지한다.
- 전체 데이터가 아니라 검색된 최대 5건만 모델 입력에 포함한다.
- 후속 질문 문맥은 애플리케이션이 제한된 최근 메시지로 관리한다. OpenAI Conversations API는 이번 범위에 추가하지 않는다.
- 정상적인 지원 범위 질문은 기본적으로 LLM을 호출한다. 검색 기반 답변은 장애 복구용으로만 사용한다.
- 답변은 제공된 검색 결과 밖의 사실을 추측하지 않는다.

## UI 동작

- 전송 중에는 입력과 빠른 질문 버튼을 비활성화한다.
- 15초 제한시간을 넘기면 로딩 애니메이션을 종료한다.
- 오류 영역에 `다시 시도` 버튼을 표시하고 마지막 실패 질문을 재사용한다.
- 사용자가 창을 닫아도 진행 중 요청은 유지하되, 컴포넌트가 해제되면 취소한다.
- 메시지는 최근 50개까지만 DOM에 유지한다.
- 후속 질문이 직전 결과를 사용한 경우 답변 근거 카드를 다시 표시한다.
- 프로젝트 관련 일상대화는 근거 카드 없이 일반 메시지로 표시한다.
- 스크린리더에는 로딩 시작, 성공, 실패 상태가 중복 낭독되지 않도록 별도의 상태 영역을 사용한다.

## 테스트 전략

### 검색 단위 테스트

다음 질의가 결과를 반환하고 의도에 맞는 필터를 적용해야 한다.

- `9월에 축제 알려줘`
- `무료인 축제 알려줘`
- `종로구에서 열리는 축제`
- `2026년 9월 축제`
- `내일 열리는 축제`
- `이번 주 축제`
- `다음 달 무료 축제`

연도·월 경계, 윤년, 서울 시간 자정 경계도 고정 날짜로 검증한다.

### Function 테스트

- 문맥 메시지 개수와 길이 제한
- 허용되지 않은 역할 거부
- `sourceIds` 최대 개수 제한
- 검색 결과가 없을 때 OpenAI 미호출
- OpenAI 시간 초과와 오류 시 fallback 반환
- 응답 모드와 오류 코드 일관성

### UI 테스트

- 15초 시간 초과 후 로딩 종료
- 실패 질문 재시도
- Enter 전송과 Shift+Enter 줄바꿈
- 중복 전송 방지
- 컴포넌트 해제 시 요청 취소
- 메시지 50개 제한

### 대표 질의 평가

정확성, 근거 일치, 평균 지연시간, 입력·출력 토큰을 기록한다. `reasoning.effort`와 `max_output_tokens` 변경은 대표 질의 결과가 기존 기준보다 나빠지지 않을 때만 적용한다.

## 단계별 완료 기준

### 1단계 완료

- 조사·날짜 표현 회귀 테스트가 통과한다.
- 후속 질문이 직전 검색 결과를 올바르게 좁힌다.
- 요청이 15초를 넘으면 UI가 복구된다.
- 실패 질문을 한 번의 동작으로 재시도할 수 있다.
- 기존 검색·Function·빌드 검증이 함께 통과한다.
- 대표 질의의 정확성을 유지하면서 토큰 또는 지연시간이 감소한다.

### 2단계 완료

- 제공된 7개 JSON 유형과 커뮤니티 게시글을 공통 계약으로 검색한다.
- 질문 유형에 맞는 공급자만 선택한다.
- LLM에는 관련 후보 최대 5건만 전달한다.
- 화면에 데이터 유형별 근거 카드가 표시된다.
- 음식점 데이터 누락 상태가 사용자와 문서에 명시된다.
- 프로젝트 관련 인사·감사·여행 취향 대화에 자연스럽게 응답한다.
- 프로젝트와 무관하거나 고위험인 요청은 지원 범위를 안내한다.
- 공급자 선택으로 불필요한 전체 데이터 검색을 수행하지 않는다.

## 제외 범위

- API 키의 브라우저 노출
- OpenAI Conversations API 기반 영구 대화 저장
- 벡터 데이터베이스
- 사용자 계정 및 서버 세션
- 음식점 데이터의 임의 수집
- 1차 최적화 단계의 스트리밍 응답
- 지도·커뮤니티 구조의 전면 재작성
